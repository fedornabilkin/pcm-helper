# План оптимизации по результатам аудита

## Цель

Повысить стабильность, поддерживаемость и производительность PCM Helper без изменения пользовательского поведения первой итерацией.

## Контекст

Аудит выполнен 2026-07-01. Проект представляет собой Vue 3/Vite приложение в директории `app` с двумя доменными зонами: PCM-справочник и networker-граф на D3. Производственная сборка `npm run build` успешно проходит при запуске вне sandbox.

## Наблюдения

- В `package.json` скрипт `test:unit` ссылается на `vue-cli-service`, но проект использует Vite и не содержит Vue CLI в зависимостях.
- В `src/router.js` и части стартовых данных видны поврежденные русские строки, похожие на mojibake.
- CSS-бандл сборки около 764 KB до gzip из-за полного подключения Bulma и Font Awesome.
- TypeScript включен в строгом режиме, но проект смешивает `.js`, `.ts` и импорты с явным расширением `.ts`.
- В `localStore.ts` и импорте файлов есть прямой `JSON.parse` без обработки ошибок.
- В D3-слое используется `@ts-ignore`, много `any`, а часть callback-типов не соответствует фактическим данным.
- В `graphStore.ts` демо-данные содержат повторяющийся `id` для functional circles.
- В `DrawNetwork.drawTick` обращение к `this.scope[d.id].circle` может упасть, если круг связан с отсутствующим узлом.
- Сервис PCM возвращает статические JSON-данные через ручные Promise-обертки, хотя операции синхронные.
- Нет зафиксированных правил по форматированию, линтингу, тестам и структуре планов.

## Актуальный порядок работ

1. **Сохранность данных:** D4, затем D3 и проверки D5–D7.
2. **Надёжность графа:** B7–B8 после проверки реальных сценариев хранения.
3. **Производительность:** C2–C8 отдельным этапом, после стабилизации поведения.

Пункты A1, D1 и D2 закрыты последующими изменениями: маршруты переведены на TypeScript с корректными заголовками, а в импорте появились версия схемы и валидация до применения.
