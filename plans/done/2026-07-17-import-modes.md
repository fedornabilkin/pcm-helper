# План внедрения режимов импорта данных

## Цель

Добавить два безопасных режима импорта сетей:

1. Полная замена текущих данных содержимым файла.
2. Выборочное Git-подобное объединение с выбором контактов и разрешением конфликтов.

Текущая реализация `GraphService.import()` сразу вызывает `fromDTO()` и заменяет коллекции. Перед развитием импорта этот процесс необходимо разделить на чтение, анализ и применение.

## Режимы

| Режим | Поведение |
|---|---|
| Заменить данные | Полностью заменяет текущие узлы, связи, теги и функциональные круги |
| Объединить данные | Сравнивает графы, позволяет выбрать контакты и разрешить конфликты |

## Этап 1. Версионирование и постоянные идентификаторы

- [x] Добавить `schemaVersion` экспортируемого файла отдельно от версии приложения.
- [x] Добавить постоянный `uid` контактам.
- [x] Добавить постоянные `uid` связям, тегам и функциональным кругам.
- [x] Оставить числовые `id` локальными идентификаторами, которые можно переназначать при импорте.
- [x] Для старых файлов без `uid` искать возможные совпадения по нормализованному имени.
- [x] Не объединять найденные по имени совпадения автоматически без подтверждения пользователя.

## Этап 2. Безопасный конвейер импорта

Разделить импорт на три операции:

1. `parse` — чтение и валидация файла без изменения графа.
2. `analyze` — сравнение локальных и импортируемых данных.
3. `apply` — атомарное применение сформированного плана.

- [x] Добавить валидацию структуры файла до создания сущностей.
- [x] Не изменять текущий граф до вызова `apply`.
- [x] Создавать резервный снимок текущего состояния перед применением.
- [x] При любой ошибке восстанавливать исходное состояние.

## Этап 3. Режим полной замены

- [x] Показывать количество импортируемых контактов, связей, тегов и кругов.
- [x] Показывать предупреждение о безвозвратной потере текущих данных.
- [x] Требовать явное подтверждение.
- [x] Автоматически сохранять локальный резервный снимок.
- [x] Заменять все коллекции одной операцией.
- [x] Сохранять и перерисовывать граф только после успешной проверки результата.

## Этап 4. Экран выборочного объединения

После чтения файла показывать контакты со статусами:

- `Новый` — отсутствует в текущей сети.
- `Без изменений` — локальные и входящие данные совпадают.
- `Изменён` — входящая версия отличается.
- `Конфликт` — контакт изменён с обеих сторон.
- `Возможный дубль` — совпадение старого формата найдено по имени.

Добавить:

- [x] Выбор отдельных контактов.
- [x] Действие «Выбрать все новые».
- [x] Действие «Выбрать всё».
- [x] Поиск по контактам.
- [x] Фильтры по статусам.
- [x] Счётчик выбранных контактов.
- [x] Предварительный итог создаваемых и обновляемых сущностей.

## Этап 5. Разрешение конфликтов

Для конфликтного контакта предоставить варианты:

- оставить текущую версию;
- использовать импортируемую версию;
- объединить данные по отдельным полям.

Поля для ручного разрешения:

- [x] Имя.
- [x] Описание.
- [x] Тип контакта.
- [x] PCM-данные.
- [x] Факты.
- [x] Теги.
- [x] Координаты и фиксация на графе.

Теги и факты можно объединять как множества. Для существующего контакта по умолчанию следует сохранять локальные координаты.

## Этап 6. Зависимые сущности

- [x] Импортировать факты вместе с выбранным контактом.
- [x] Создавать отсутствующие теги выбранных контактов.
- [x] Импортировать связь, только если оба её контакта существуют после объединения.
- [x] Если второй контакт не выбран, предлагать выбрать его либо пропустить связь.
- [x] Импортировать функциональный круг только вместе с существующим узлом-владельцем.
- [x] Переназначать числовые идентификаторы без коллизий.
- [x] Сохранять таблицы соответствия старых и новых идентификаторов для всех зависимостей.

Перед применением показывать итоговое количество:

- новых и обновлённых контактов;
- создаваемых и пропускаемых связей;
- добавляемых тегов;
- добавляемых функциональных кругов.

## Этап 7. Git-подобное трёхстороннее слияние

Для полноценного трёхстороннего слияния использовать:

- `base` — общую исходную версию;
- `local` — текущие данные;
- `incoming` — импортируемые данные.

Правила:

| Состояние | Результат |
|---|---|
| Изменён только `incoming` | Принять входящее изменение |
| Изменён только `local` | Оставить локальное изменение |
| Обе стороны изменены одинаково | Принять автоматически |
| Обе стороны изменены по-разному | Показать конфликт |
| Удаление с одной стороны и изменение с другой | Показать конфликт |

- [x] Добавить в экспорт `networkUid`.
- [x] Добавить `revisionId`.
- [x] Добавить `parentRevisionId`.
- [x] Хранить базовый снимок или доступную историю снимков.
- [x] При отсутствии общего предка переходить к двухстороннему сравнению с ручным разрешением.

## Этап 8. Архитектура

Добавить отдельный сервис анализа и применения импорта:

```ts
class GraphImportService {
  analyze(
    local: DataTransfer,
    incoming: DataTransfer,
    base?: DataTransfer,
  ): ImportAnalysis

  buildPlan(
    analysis: ImportAnalysis,
    selection: ImportSelection,
    resolutions: ConflictResolution[],
  ): ImportPlan

  apply(local: DataTransfer, plan: ImportPlan): DataTransfer
}
```

Основные модели:

- `ImportMode`
- `ImportAnalysis`
- `ImportCandidate`
- `ImportStatus`
- `ImportSelection`
- `ConflictResolution`
- `ImportPlan`
- `ImportResult`

`GraphService` должен получать уже сформированный и проверенный результат, а не самостоятельно разрешать конфликты.

## Проверки

- [x] Совпадающие числовые `id` в двух графах не создают коллизий.
- [x] Дубли контактов по имени требуют подтверждения.
- [x] Можно импортировать только часть контактов.
- [x] Связь с невыбранным контактом не создаётся автоматически.
- [x] Конфликты тегов и фактов разрешаются предсказуемо.
- [x] Удаление против изменения определяется как конфликт.
- [x] Старый формат без `uid` импортируется безопасно.
- [x] Отмена на любом этапе не изменяет данные.
- [x] Ошибка применения приводит к полному откату.
- [x] После объединения нет ссылок на отсутствующие сущности.

## Рекомендуемая последовательность

1. Версионирование схемы, постоянные идентификаторы и валидация.
2. Безопасный режим полной замены.
3. Двухсторонний выборочный импорт.
4. Разрешение конфликтов по полям.
5. История ревизий и полноценное трёхстороннее слияние.

## Критерии готовности

- [x] Импорт не изменяет граф до финального подтверждения.
- [x] Режим замены атомарен и поддерживает откат.
- [x] Пользователь может выбрать конкретные импортируемые контакты.
- [x] Зависимые сущности обрабатываются без битых ссылок.
- [x] Все конфликты явно показаны и разрешены до применения.
- [x] Старые файлы поддерживаются через безопасную миграцию.
