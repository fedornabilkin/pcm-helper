# План внедрения Premium-доступа

## Цель

На первом этапе добавить сессионный Premium-доступ по промокоду `premium`, централизовать ограничения тарифов и подготовить архитектуру к будущей серверной авторизации.

Premium должен сохраняться при обновлении страницы в текущей вкладке и сбрасываться после завершения браузерной сессии.

## Предлагаемые ограничения

| Возможность | Базовый | Premium |
|---|---:|---:|
| Сети | 1, включая «Основную» | 3 |
| Контакты в сети | 50 | 500 |
| Связи | Без отдельного лимита | Без отдельного лимита |
| Пользовательские теги | 5 | 50 |
| Поиск и подсветка | Доступно | Доступно |
| Экспорт своих данных | Доступно | Доступно |
| Импорт с заменой | Доступно | Доступно |
| Выборочное слияние при импорте | Недоступно | Доступно |
| AI-анализ PCM | Недоступно | Доступно |
| Стандартные функциональные круги | Доступно | Доступно |
| Произвольные круги и настройки | Недоступно | Доступно |
| Расширенная статистика и фильтры | Базовая | Полная |

Экспорт пользовательских данных должен оставаться доступным на обоих тарифах.

## Этап 1. Единая модель доступа

Добавить уровни доступа:

```ts
type AccessTier = 'basic' | 'premium'
```

Создать единый composable или сервис `usePremiumAccess`:

```ts
interface PremiumAccess {
  isPremium: Readonly<Ref<boolean>>
  tier: Readonly<Ref<AccessTier>>
  activate(code: string): PremiumActivationResult
  deactivate(): void
  can(feature: FeatureCode): boolean
  getLimit(resource: LimitCode): number
}
```

- [ ] Не размещать проверки `isPremium` напрямую во множестве компонентов.
- [ ] Все разрешения и лимиты получать через один слой доступа.
- [ ] Описать типы функциональных разрешений и лимитов.

## Этап 2. Активация промокодом

- [ ] Добавить модалку «Premium-доступ».
- [ ] Добавить поле промокода и кнопку активации.
- [ ] Принимать `premium` без учёта регистра и окружающих пробелов.
- [ ] При успешной активации сохранять только состояние доступа.
- [ ] Не сохранять введённый промокод.
- [ ] Показывать статус «Premium активен до конца текущей сессии».
- [ ] Показывать ошибку при неправильном коде без закрытия модалки.

Сессионное состояние:

```ts
sessionStorage.setItem('pcm-premium-access', 'active')
```

При инициализации приложения сервис должен прочитать этот ключ и восстановить состояние.

## Этап 3. Централизованная политика тарифов

Создать конфигурацию:

```ts
const ACCESS_POLICY = {
  basic: {
    networks: 1,
    nodesPerNetwork: 50,
    tagsPerNetwork: 5,
    aiAnalysis: false,
    selectiveImport: false,
    customFunctionalCircles: false,
    advancedStatistics: false,
  },
  premium: {
    networks: 3,
    nodesPerNetwork: 500,
    tagsPerNetwork: 50,
    aiAnalysis: true,
    selectiveImport: true,
    customFunctionalCircles: true,
    advancedStatistics: true,
  },
} as const
```

- [ ] Заменить текущий константный лимит сетей получением значения из политики.
- [ ] Подключить лимит контактов.
- [ ] Подключить лимит тегов.
- [ ] Описать Premium-разрешения для AI и будущего выборочного импорта.

## Этап 4. Проверки на двух уровнях

Ограничения должны проверяться:

1. В интерфейсе — скрытие, блокировка или объяснение недоступного действия.
2. В сервисах — защита от обхода через прямой вызов метода.

Сервисы должны возвращать структурированный результат:

```ts
type AccessGuardResult =
  | {success: true}
  | {
      success: false
      reason: 'premium_required' | 'limit_reached'
      limit?: number
    }
```

- [ ] Защитить `NetworkService.addNetwork()`.
- [ ] Защитить `GraphService.addNode()`.
- [ ] Защитить создание тегов.
- [ ] Защитить AI-анализ.
- [ ] Защитить будущий выборочный импорт.

## Этап 5. UX ограничений

При достижении числового лимита:

- [ ] Скрывать или блокировать действие создания.
- [ ] Показывать метрику в формате `50 / 50`.
- [ ] Предоставлять понятный переход к Premium-модалке.

Для Premium-функций:

- [ ] Показывать значок замка.
- [ ] По нажатию открывать Premium-модалку.
- [ ] Кратко объяснять преимущество Premium.
- [ ] Не скрывать полностью значимые возможности, если пользователю важно узнать об их существовании.

После активации ограничения должны сниматься сразу, без перезагрузки страницы.

## Этап 6. Работа с существующими данными

Если пользователь уже превысил базовый лимит:

- [ ] Не удалять существующие данные.
- [ ] Разрешить просмотр и редактирование.
- [ ] Разрешить удаление.
- [ ] Разрешить экспорт.
- [ ] Блокировать только создание новых сущностей сверх лимита.
- [ ] После Premium-активации сразу разрешать создание в пределах Premium-лимита.

## Этап 7. Точки внедрения

Подключить политику доступа к:

- [ ] Созданию сети.
- [ ] Добавлению контакта.
- [ ] Созданию тегов.
- [ ] AI-анализу PCM.
- [ ] Будущему выборочному импорту.
- [ ] Пользовательским функциональным кругам.
- [ ] Расширенным метрикам и фильтрам.
- [ ] Блоку управления сетью.

## Этап 8. Компоненты интерфейса

- [ ] Кнопка или бейдж Premium в интерфейсе.
- [ ] Модалка ввода промокода.
- [ ] Сообщение об успешной активации.
- [ ] Сообщение о неправильном промокоде.
- [ ] Компонент объяснения ограничения.
- [ ] Значок Premium рядом с закрытыми возможностями.
- [ ] Индикатор активного Premium-доступа.

## Этап 9. Проверки

- [ ] Новая сессия начинается в базовом режиме.
- [ ] `premium`, ` Premium ` и `PREMIUM` активируют доступ.
- [ ] Неправильный код не меняет состояние.
- [ ] Обновление страницы сохраняет Premium.
- [ ] Новая браузерная сессия сбрасывает Premium.
- [ ] Лимиты снимаются сразу после активации.
- [ ] Сервисные проверки нельзя обойти через прямой вызов UI-метода.
- [ ] Существующие данные сверх лимита сохраняются.
- [ ] Экспорт доступен на обоих уровнях.
- [ ] Интерфейс работает на мобильном экране.
- [ ] Интерфейс корректно выглядит в тёмной теме.

## Этап 10. Подготовка к серверной модели

Frontend-промокод не является настоящей защитой: его можно обнаружить в JavaScript или обойти через DevTools.

Для первого этапа этого достаточно для проверки UX и продуктовых ограничений. Перед подключением оплаты необходимо:

- [ ] Перенести проверку промокода на сервер.
- [ ] Добавить пользовательские аккаунты.
- [ ] Выдавать подписанный токен с тарифом и сроком действия.
- [ ] Проверять платные серверные операции на backend.
- [ ] Не доверять Premium-флагу из frontend-хранилища.
- [ ] Добавить срок действия, отзыв и обновление подписки.

## Рекомендуемая последовательность

1. Модель доступа и тарифная политика.
2. Сессионная активация по промокоду.
3. Premium-модалка и индикатор статуса.
4. Лимит сетей и контактов.
5. Лимит тегов и блокировка AI.
6. Подключение выборочного импорта и расширенных функций.
7. Серверная авторизация перед запуском оплаты.

## Критерии готовности первого этапа

- [ ] Промокод `premium` активирует Premium до конца сессии.
- [ ] Состояние сохраняется в `sessionStorage`.
- [ ] Все ограничения определяются централизованной политикой.
- [ ] Ограничения проверяются в UI и сервисах.
- [ ] Существующие пользовательские данные не удаляются.
- [ ] Экспорт доступен независимо от тарифа.
- [ ] Смена уровня доступа применяется без перезагрузки.
