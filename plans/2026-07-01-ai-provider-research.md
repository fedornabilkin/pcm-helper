# План исследования AI-провайдеров без developer token

## Цель

Выбрать запасной или основной AI-провайдер для определения PCM-типа по тексту, если Pollinations anonymous/referrer окажется нестабильным или недостаточным по лимитам.

## Контекст

Текущий вариант использует Pollinations OpenAI-compatible endpoint с `referrer`. Второй кандидат - Puter.js, где не нужен developer API key, но пользовательская модель оплаты и авторизации отличается от обычного публичного API.

## Группа A: Puter.js

- [ ] A1. Проверить официальную документацию Puter.js AI Chat и зафиксировать способ подключения в Vue/Vite.
- [ ] A2. Проверить модель оплаты Puter.js user-pays и описать UX-последствия для пользователя.
- [ ] A3. Создать минимальный эксперимент Puter.js вне production UI и получить ответ на тестовый PCM-промпт.
- [ ] A4. Проверить, требуется ли пользовательская авторизация Puter.js перед первым AI-запросом.
- [ ] A5. Оценить, можно ли использовать Puter.js без хранения ключей, токенов и персональных данных в приложении.

## Группа B: Альтернативные сервисы без developer token

- [ ] B1. Проверить WebLLM как локальный браузерный вариант без внешнего API.
- [ ] B2. Проверить ограничения WebLLM по WebGPU, размеру моделей и времени первого запуска.
- [ ] B3. Проверить Hugging Face Inference Providers на наличие реально доступного режима без token для browser-only приложения.
- [ ] B4. Проверить OpenRouter, Together, Groq и Gemini на наличие бесплатного режима и подтвердить, нужен ли developer API key.
- [ ] B5. Составить таблицу кандидатов с колонками: токен, CORS, лимиты, стоимость, качество, сложность интеграции.

## Группа C: Архитектурная подготовка

- [ ] C1. Выделить интерфейс AI-провайдера для метода определения PCM-типа по тексту.
- [ ] C2. Добавить адаптер Pollinations под общий интерфейс без изменения UI.
- [ ] C3. Подготовить адаптер-заглушку для Puter.js под общий интерфейс.
- [ ] C4. Добавить единый формат результата анализа: тип, уверенность, причины, альтернатива, резюме.
- [ ] C5. Добавить обработку ошибок провайдера с понятным текстом для пользователя.

## Группа D: Решение

- [ ] D1. Сравнить Pollinations, Puter.js и локальный WebLLM по таблице из B5.
- [ ] D2. Выбрать основной и резервный AI-провайдер для первого production-варианта.
- [ ] D3. Зафиксировать решение в `agreements` или отдельном техническом решении, если выбор влияет на архитектуру.
- [ ] D4. Сформировать отдельный план внедрения выбранного провайдера после завершения исследования.

## Критерии готовности

- Все кандидаты проверены по документации и минимальным техническим ограничениям.
- Для Puter.js подтвержден или отклонен UX с пользовательской авторизацией.
- Есть таблица сравнения провайдеров.
- Есть решение по основному и резервному провайдеру.
