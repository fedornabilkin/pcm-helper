# Архитектурное соглашение

## Назначение

Это соглашение фиксирует правила проектирования нового кода в PCM Helper. Его применять ко всем новым функциональным изменениям и учитывать при рефакторинге существующих модулей.

## Файлы и язык

- Все новые файлы с логикой создавать в формате `.ts`.
- Новые JavaScript-файлы не создавать без отдельного зафиксированного решения.
- Для Vue-компонентов использовать `<script setup lang="ts">`.
- В разметке Vue-компонентов использовать Pug: `<template lang="pug">`.
- Типы, интерфейсы и контракты выносить в отдельные `.ts` файлы, если они используются более чем в одном месте или усложняют чтение компонента.

## Vue-компоненты

- Vue-компоненты должны быть тонким UI-слоем.
- В компонентах Vue использовать минимум JS/TS-кода.
- Для разметки, сеток, форм, кнопок, панелей и типовых состояний UI в первую очередь использовать Bulma.
- Бизнес-логику, вычисления, правила преобразования данных и сценарии работы выносить из компонентов в классы, сервисы, builders, composables или отдельные доменные файлы.
- Компонент должен отвечать за отображение, связывание UI-событий и передачу данных в доменный слой.
- Новые UI-элементы проектировать как переиспользуемые компоненты, если есть хотя бы два ожидаемых места применения или повторяется структура/поведение.
- Переиспользуемые компоненты размещать ближе к домену, где они используются, либо в общем слое только при реальной междоменной пользе.

## Доменная модель

- В приложении использовать объекты сущностей, а не работать напрямую с сырыми JSON-данными в UI и сервисах.
- Сырые данные отделять от объектов сущностей слоем builders.
- Builder отвечает за создание доменного объекта из DTO/JSON и за восстановление коллекций.
- Entity отвечает за доменное состояние и поведение конкретной сущности.
- Service отвечает за сценарии работы, координацию сущностей, хранилища, импорта/экспорта и внешних адаптеров.
- DTO использовать как формат передачи и хранения данных, не смешивая его с поведением entity.

## Паттерны и принципы

- Приоритет отдавать понятным и уместным паттернам GoF, если они уменьшают связность или упрощают расширение.
- Не внедрять паттерн ради паттерна: решение должно быть проще сопровождать, чем прямолинейная реализация.
- Соблюдать SOLID при проектировании классов, сервисов и builders.
- Соблюдать GRASP при распределении ответственностей между сущностями, сервисами, builders и UI.
- Предпочитать композицию наследованию, если наследование не отражает устойчивую доменную иерархию.
- Зависимости направлять от UI к прикладному и доменному слоям, но не наоборот.
- Код должен быть открыт для добавления новых сущностей/типов поведения без массового изменения существующих компонентов.

## Практические правила

- Если Vue-компонент начинает содержать сложные условия, преобразования коллекций или работу с несколькими сущностями, эту логику нужно вынести.
- Если данные приходят из JSON, `localStorage` или файла, сначала преобразовать их в DTO, затем через builder в entity.
- Если entity создается более чем в одном месте, создание должно проходить через builder.
- Если сервис зависит от способа хранения или формата файла, зависимость оформлять через adapter или отдельный слой transfer.
- Если новый код нарушает это соглашение из-за ограничения текущей архитектуры, добавить задачу в план рефакторинга.
